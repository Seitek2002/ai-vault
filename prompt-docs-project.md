---

## Роль и цель

Ты — старший full-stack инженер, реализующий веб-приложение для **хранения, импорта, редактирования и экспорта деловых документов**. Ключевая особенность: документы может редактировать не только человек, но и **ИИ**, причём все внесённые ИИ изменения **визуально выделяются другим цветом** (стиль «отслеживания изменений», как track changes в Word), чтобы пользователь сразу видел, что именно поправил ИИ, и мог принять или отклонить каждую правку.

Работай итеративно: сначала согласуй архитектуру и модель данных, затем строй MVP по этапам (см. раздел «Этапы»). Не пиши весь проект одним куском — показывай прогресс и спрашивай, если встречаешь неоднозначность.

## Типы документов

Система работает с пятью типами деловых документов (СНГ-формат):

1. **КП** — коммерческое предложение (шапка компании, клиент, перечень товаров/услуг с ценами, итоги, срок действия, условия).
2. **Договор** — с заказчиками и клиентами (стороны, предмет, условия, оплата, ответственность, реквизиты, подписи).
3. **Счёт-фактура** — налоговый документ (продавец/покупатель, позиции, НДС, итоги).
4. **Счёт на оплату** — реквизиты получателя, позиции, сумма к оплате.
5. **АВР** — акт выполненных работ (стороны, перечень работ, суммы, подписи).

Каждый тип имеет **структурированные поля** (метаданные: контрагент, суммы, даты, номер) **и редактируемое тело** документа (rich text). Заложи шаблоны (templates) для каждого типа: предзаполненная структура + поля, которые пользователь/ИИ заполняет. Шаблоны должны легко расширяться (в будущем добавятся новые типы).

## Технологический стек

### Frontend (зафиксировано)
- **Next.js (App Router) + TypeScript**
- **TailwindCSS** — стилизация
- **Zustand** — клиентское состояние (UI, черновики, состояние редактора)
- **TanStack Query** — серверное состояние, кэш, мутации, синхронизация с API
- **Rich-text редактор:** TipTap (на базе ProseMirror) — у него хорошая поддержка `marks`/`decorations`, что критично для подсветки правок ИИ и accept/reject изменений. **[РЕШЕНИЕ: TipTap по умолчанию; альтернатива — Lexical]**

### Backend (собственный, по требованию заказчика)
- **Node.js + NestJS** — модульная архитектура, DI, удобно делить на модули (auth, documents, files, ai, export). **[РЕШЕНИЕ: NestJS по умолчанию; альтернатива — Express, если нужен более лёгкий вариант]**
- **PostgreSQL + Prisma** — основная БД (документы, версии, пользователи, контрагенты). **[РЕШЕНИЕ: Prisma по умолчанию; альтернатива — TypeORM]**
- **S3-совместимое хранилище файлов** — оригиналы импортированных файлов, экспортированные PDF/DOCX, фото. Локально — MinIO, в проде — любой S3-провайдер.
- **Авторизация:** JWT (access + refresh) с ролями (RBAC), поддержка **нескольких пользователей**. Минимум роли: `admin`, `manager`. На старте — email + пароль; OAuth — позже.

### Структура репозитория
Монорепозиторий на **pnpm workspaces (+ опционально Turborepo)**:
```
/apps/web      — Next.js фронтенд
/apps/api      — NestJS бэкенд
/packages/types — общие TypeScript-типы (DTO, модели документов)
```
Общие типы документов и API-контракты держи в `/packages/types`, чтобы фронт и бэк не расходились.

## Интеграция с ИИ

- Провайдер ИИ **пока не выбран** — спроектируй слой как **провайдер-независимый адаптер** (интерфейс `AiProvider`), чтобы можно было подключить любой бэкенд/API позже без переписывания логики. **[РЕШЕНИЕ: выбор провайдера отложен]**
- Заказчик планирует подключать **внешний (чужой) бэкенд ИИ** — предусмотри конфигурацию через переменные окружения (base URL, ключ, модель) и чёткую границу: бизнес-логика приложения не должна зависеть от конкретного провайдера.
- Основные ИИ-операции, которые нужно поддержать через этот адаптер:
  1. **Редактирование/доработка текста документа** (вход: текущий текст + инструкция пользователя; выход: новый текст + **структурированный diff**).
  2. **OCR / распознавание текста** из импортированных фото и сканов (vision-модель или Tesseract — см. «Импорт»).
  3. (Опционально, позже) генерация документа из краткого описания по шаблону.

### Подсветка изменений ИИ (ключевая фича)
- Когда ИИ редактирует документ, не подменяй текст молча. Вычисляй **diff** между исходным и новым текстом (например, через `diff-match-patch` или ProseMirror-уровневый diff) и рендери его как **track-changes**:
  - **Вставки ИИ** — выделить (например, фоновой подсветкой/подчёркиванием).
  - **Удаления ИИ** — показать зачёркнутым.
  - Каждое изменение должно иметь кнопки **«Принять» / «Отклонить»** (по отдельности и «принять все»).
- **Палитра подсветки** (важно — не сливаться с акцентом темы `#87CEEB`):
  - Вставки: мягкий зелёный (`#4ADE80` / подсветка `rgba(74,222,128,0.15)`).
  - Удаления: красный + зачёркивание (`#F87171`).
  - Общая пометка «изменено ИИ»: янтарный (`#FBBF24`).
  - **[РЕШЕНИЕ: финальные оттенки подбери под контраст на тёмном фоне; голубой `#87CEEB` оставь только для акцентов интерфейса, не для diff]**
- Храни правки ИИ как отдельный слой/состояние до подтверждения, чтобы можно было откатить.

## Импорт и экспорт

### Импорт
- Принимай **файлы любых расширений**. Поведение по типу:
  - **PDF / DOCX** → извлечь текст и (по возможности) структуру → поместить в редактируемое тело документа.
  - **PNG / JPG / сканы / фото документов** → прогнать через **OCR** (vision-модель ИИ-адаптера или Tesseract.js) → получить редактируемый текст. **[РЕШЕНИЕ: метод OCR — через ИИ-провайдера или Tesseract — выбери на этапе реализации]**
  - **Прочие/неизвестные расширения** → сохранить как вложение-оригинал, дать скачать; пытаться распознать, если возможно.
- Всегда **сохраняй оригинал файла** в хранилище (S3), даже после OCR/конвертации — чтобы был первоисточник.

### Экспорт
- **PDF** — серверная генерация для точной вёрстки делового документа (Puppeteer/Playwright по HTML-шаблону). **[РЕШЕНИЕ]**
- **Word (.docx)** — через библиотеку `docx`.
- **Скачивание оригинала** импортированного файла.
- Экспорт должен учитывать реквизиты/шаблон конкретного типа документа (КП, договор и т.д.).

## Модель данных (ориентир, уточни в Prisma-схеме)

- `User` — пользователи, роли.
- `Counterparty` — контрагенты (заказчики/клиенты): название, реквизиты, контакты.
- `Document` — тип (enum: KP | CONTRACT | INVOICE_FACTURA | INVOICE_PAYMENT | AVR), номер, статус (draft / final / sent / signed), контрагент, метаданные-поля (JSON по типу), тело (rich-text / ProseMirror JSON), даты.
- `DocumentVersion` — версионирование (важно: ИИ-правки → новые версии; история изменений).
- `DocumentChange` — отдельные правки ИИ для accept/reject (тип: insert/delete, позиция, автор=AI/USER, статус).
- `FileAsset` — оригиналы и экспортированные файлы (ссылка на S3, mime, размер).

Веди **версионирование** и **аудит** (кто/что/когда менял, человек или ИИ).

## UI / UX

- **Тёмная тема** с акцентным цветом **`#87CEEB`** (небесно-голубой) — кнопки, активные состояния, ссылки, фокус. Фон — глубокий тёмный (например, `#0F172A`/`#111827`), поверхности — на ступень светлее. Текст — высокий контраст.
- Задай дизайн-токены (CSS-переменные / Tailwind theme): `--accent: #87CEEB`, фоновые слои, цвета diff (см. палитру выше).
- **Адаптивность обязательна — ПК и мобильные.** Mobile-first. На десктопе — двухпанельный режим (список/документ или документ/ИИ-чат), на мобильных — стек/вкладки. Редактор и панель ИИ-правок должны быть удобны на узком экране.
- Основные экраны: вход; список документов с фильтрами (по типу, контрагенту, статусу, дате) и поиском; страница документа (редактор + поля + панель ИИ); импорт; настройки/реквизиты компании.

## Безопасность

- JWT (access + refresh), хеширование паролей (argon2/bcrypt).
- RBAC на уровне эндпоинтов.
- Валидация входных данных (class-validator / zod), защита загрузки файлов (проверка размера, mime, антивирус-хук — опционально).
- Не логировать секреты; ключи ИИ-провайдера и S3 — только в env.

## Этапы реализации

1. **Скелет монорепо:** структура, общие типы, Tailwind с тёмной темой и токенами `#87CEEB`, базовый layout (адаптивный).
2. **Бэкенд-фундамент:** NestJS + Prisma + PostgreSQL, авторизация (JWT, роли), CRUD `Document` и `Counterparty`.
3. **Редактор документов:** TipTap, шаблоны 5 типов, структурированные поля + тело, сохранение/версии.
4. **Слой ИИ (адаптер):** интерфейс `AiProvider`, операция редактирования с возвратом diff; **подсветка правок + accept/reject** (главная фича).
5. **Импорт:** загрузка любых файлов, конвейер PDF/DOCX → текст, **OCR для png/jpg**, сохранение оригинала в S3.
6. **Экспорт:** PDF (серверный) + DOCX + скачивание оригинала.
7. **Полировка:** фильтры/поиск, мобильная адаптивность, аудит/история, обработка ошибок, состояния загрузки.

Начни с этапа 1 и согласуй со мной архитектуру и Prisma-схему, прежде чем двигаться к этапу 3.
